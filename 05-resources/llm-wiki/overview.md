# LLM Wiki 概念

## 核心思想

大多数人对 LLM 处理文档的体验是 RAG：上传一堆文件，LLM 在查询时检索相关片段，生成答案。这能跑通，但每次提问 LLM 都要从零重新挖掘知识——没有任何积累。问一个需要综合五份文档才能回答的微妙问题，LLM 得每次都临时拼凑碎片。NotebookLM、ChatGPT 文件上传、大多数 RAG 系统都是这个模式。

这里的思路不同。**LLM 增量地构建并持续维护一个持久化的 Wiki**——一套结构化、相互链接的 markdown 文件，位于你和原始资料之间。

- 你添加新资料时，LLM 不只是索引它。它阅读、提取关键信息，并整合到现有 Wiki 中——更新实体页面、修订主题摘要、标注新旧矛盾、强化或修正既有认知。
- **知识被编译一次，然后持续保鲜**，而非每次查询时重新推导。

核心区别：**Wiki 是一个持续复利的产物。** 交叉引用已经建好，矛盾已经标注，综合结论已经反映了你读过的一切。每添加一个资料、每问一个问题，Wiki 都在变厚。

你几乎不需要自己写 Wiki —— LLM 负责所有编写和维护。你的职责是：筛选资料、引导探索、问对问题。LLM 干所有苦活：摘要、交叉引用、归档、记账。

类比：Obsidian 是 IDE，LLM 是程序员，Wiki 是代码库。

## 三步结构

| 步骤 | 文件夹 | 说明 |
|------|--------|------|
| 第一步 | **raw/** | 你精选的源文档。文章、论文、截图、数据。只读，LLM 从不修改。你的真相来源。 |
| 第二步 | **wiki/** | LLM 生成的结构化知识文章。摘要页、实体页、概念页、对比页、综述页。LLM 全权维护。 |
| 第三步 | **outputs/** | AI 回答问题时生成的报告、分析、草稿。好的回答不回丢，单独存放，也可升格为 wiki 页面。 |

## 维护配置

**规范 (schema.md)** 告诉 LLM Wiki 的结构、约定、工作流。这是让 LLM 成为有纪律的 Wiki 维护者而非通用 chatbot 的关键配置。你和 LLM 在使用中共同演化这份文件。

## 三类操作

**收录 (Ingest)**：你放入新资料到 raw/ → LLM 阅读 → 和你讨论要点 → 写入 wiki/ → 更新索引 → 追加操作日志。一份资料可能触发 10-15 个页面的更新。

**查询 (Query)**：你问问题 → LLM 搜索 wiki/ → 阅读后综合回答（带引用）。好的回答可以**保存到 outputs/** —— 你的探索也会在知识库中复利。

**体检 (Lint)**：定期让 LLM 做健康检查——页面矛盾、过时论断、孤儿页面、缺失的交叉引用、重要概念缺少专属页面。

## 为什么有效

维护知识库最烦人的不是阅读或思考，而是记账——更新交叉引用、保持摘要同步、标注新旧矛盾、跨几十个页面保持一致性。人类放弃 Wiki 是因为维护成本超过价值增长。LLM 不会无聊、不会忘、一次能触及 15 个文件。维护成本接近零，Wiki 就一直活着。

人类的任务：筛选资料、引导分析、问好问题、思考含义。LLM 的任务：剩下的一切。
