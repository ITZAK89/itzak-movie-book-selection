1. **KRDict API 管道验证**：构建了「AI种子词表 → KRDict API查询官方释义 → AI翻译为中文 → 写入HTML」管道。100词测试成功率87%，API对6级术语和惯用语覆盖率有限（~56%）。

2. **数据结构迭代**：经过三轮调整——纯AI例句 → defs+ex分离 → senses多义词义项分组。

3. **词汇量扩充到459词**：100词KRDict增强词 + 359词AI全生成（6级占71%），覆盖14分类。

## 第三部分：重构与排错（05-21 晚间 - 05-22 凌晨）

### 核心要点

1. **大规模代码丢失**：在 Shadowing 板块多次 Edit 操作中，关键函数（`loadReviewList`、`renderChips`、`flipCard` 等）、状态变量、事件绑定被逐步误删，仅剩数据数组骨架，页面完全瘫痪。

2. **全量重建**：放弃修补，用 `regenerate_html.py` 从 JSON 数据文件（batch*.json、grammar_*.json）全新组装 HTML/JS。一次生成，一次修正（cat-filters → dp-cat），页面恢复。

3. **经验教训固化**：超过 100KB 的 HTML 大文件不反复 Edit，用脚本从数据源重新生成；每次大改动后必须在浏览器实测，不依赖 `node --check`；改前备份。已存入 memory 系统供后续会话参考。

## 第四部分：Shadowing 板块重建（05-22 凌晨）

### 核心要点

1. **架构切换**：从「内置10个视频」改为「用户自供内容」——粘贴 YouTube 链接 + VTT 字幕，即开即用。

2. **自动音频下载**：自定义 Python HTTP 服务器（`server.py`）提供 `/api/download` 端点，接收 YouTube URL → yt-dlp 下载音频 → 返回本地路径。

3. **Echo Method 界面**：Reveal/Conceal 显示韩文字幕、Prev/Next 翻句、点击字幕跳转、Enter/←/→ 快捷键、Load New 重新加载。

4. **音频同步修复**：添加 Range 请求支持（`206 Partial Content`）→ 解决了浏览器无法 seek 音频的关键问题。

5. **双链接访问**：`file://` 用于快速查看 Vocabulary/Grammar（无需启动），`http://localhost:8766` 用于完整功能（含 Shadowing）。

### 关键概念

- **就地重建**：不修补损坏文件，从数据源（JSON）全新生成 HTML
- **Range 请求**：HTTP 206 Partial Content，浏览器 seek 音频的必要条件
- **用户自供内容**：不内置跟读库，用户提供 YouTube 链接 + 字幕，页面负责解析和呈现

## 第五部分：Shadowing "已学习" 标记（05-22）

1. **Mark Learned 按钮**：在 Echo 控制栏增加可切换按钮，点击将当前字幕内容存入 `localStorage`（key: `shadow-learned`），再点击取消标记。每句字幕拼接后做 32-bit 哈希作为唯一标识。

2. **加载时重复检测**：双维度匹配——YouTube 视频 ID + 字幕内容哈希。命中时显示黄色警告条，用户可选择"仍然加载"继续。两个 bug：警告条 `display:none` 被 CSS 覆盖 → 改用 `display:block`；"仍然加载"后 `shadowData` 被清空 → 移除该行。

3. **Echo 控制栏双行布局**：第一行 Prev/Reveal/Next，第二行 Mark Learned/Load New，均居中于卡片中轴线。

## 关键概念

- **`t(key)`**：语言切换核心函数，从 `STR[LANG]` 取值，定义在脚本最开头以确保所有函数可用
- **`tCat(cn)`**：分类翻译，通过 `CAT_MAP` 查找，中文/默认语言下返回原文
- **`hashSentences()`**：32-bit 字符串哈希，拼接所有韩语句子，用于 Shadowing 内容去重
- **`recordLearningActivity()`**：按日期记录学习行为，去重排序
- **双维度重复检测**：videoId 精确匹配 + sentenceHash+sentenceCount 内容匹配
- **大文件编辑规范**：>100KB 的 HTML 用脚本从数据源重新生成，不反复 Edit；改前备份

### 后续优化建议

## 第七部分：部署准备与 GitHub Pages 上线（05-23）

### Shadowing 音频上传备用入口

为支持 GitHub Pages（无法运行 server.py 自动下载 YouTube 音频），在 Shadowing 表单增加文件上传入口：
- 保留 YouTube URL → 自动下载（本地 `python3 server.py` 可用时）
- 新增 `<input type="file">` 上传音频文件 → `URL.createObjectURL()` 创建 Blob URL → 直接播放
- 上传文件优先于路径输入
- Load New 时释放 Blob URL，清除文件选择

**遇到的 Bug**：新增第 4 个 `.shadow-label` 后，JS 按索引分配标签文本未同步更新（`sls[2]` 仍被设为 `subtitle_label` 而非 `audio_upload_label`），导致上传框标签显示为"Subtitles (VTT format)"。修复：插入 `sls[2] → audio_upload_label`，原字幕标签改为 `sls[3]`。

### Git 初始化与提交策略

- 初始化 git 仓库，`.gitignore` 排除 `.DS_Store`、`.env`、`audio/`
- **提交节奏**：编程项目（如韩语网站）改前/改后手动 commit；知识笔记以会话结束 checkpoint 为默认；CLAUDE.md / schema.md 改动需独立 commit 可追溯
- `audio/` 目录音频文件不入 git（通过 yt-dlp 按需下载）

### GitHub Pages 部署

- 单独仓库：`github.com/ITZAK89/hangul`（仅 `index.html`，数据内嵌无外部依赖）
- 网站地址：`https://itzak89.github.io/hangul/`
- Vocabulary + Grammar 全设备可用；Shadowing 通过上传音频文件可用

### 关键概念

- **Blob URL**：浏览器内将文件转为可播放的临时 URL，绕过服务器依赖
- **静态托管限制**：GitHub Pages 只能 serve 静态文件，不能执行 Python/Shell 调用 yt-dlp

## 第八部分：部署后验证与 GFW 阻断（05-23）

### 访问问题

部署后手机 5G 可访问，WiFi 不可。诊断：
- `*.github.io` 域名在中国大陆被 GFW 间歇性阻断
- 5G 有时走不同网关可通，宽带 WiFi 基本不通
- GitHub Pages API 确认站点已生效（"Your site is live at..."）

### 解决方案

- 绑定自定义域名（CNAME 绕过 `github.io` 阻断，非 100% 可靠）
- 换 Vercel / Cloudflare Pages（中国大陆连通性更好）
- 用户自有方案，暂不实施
