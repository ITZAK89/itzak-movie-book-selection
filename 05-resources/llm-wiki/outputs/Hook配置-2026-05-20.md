# 对话总结：Hook 系统全链路 + /summary 优化 + 跨机器部署

2026-05-20 当天多轮会话，从 Hook 机制诊断，到升级实施，到跨机器复制，再到 /summary 命令优化。

---

## 第一部分：Hooks 现状诊断

1. **当前 hooks 三件套存在明显缺陷**：SessionStart 只读时间戳无实质内容，PreCompact 形同虚设只写时间戳，Stop 提醒与 memory 系统脱节。
2. **PreCompact 是唯一上下文完整的时机，但它是 shell 命令无 AI 能力**；PostCompact / Stop 可以借助 Claude 的 AI 判断，但此时上下文已被压缩，信息不全。
3. **shell hook 的能力边界是核心限制**——提取关键信息需要 AI 判断力，但 hook 执行时 Claude 不参与推理，只能做机械动作。
4. **替代策略更可行**：对话中主动即时写 memory（不等压缩）、SessionStart 时由用户辅助回忆、用户手动触发"记住这个"——都比依赖自动化 hook 更可靠。

## 第二部分：策略深入讨论

1. **`/summary` 对应策略 C（手动触发）**，但保存在 `wiki/outputs/`（知识归档），而非 `~/.claude/projects/.../memory/`（跨会话自动记忆），两者目的不同。
2. **策略 A（Claude 主动写入）无硬性触发机制**，全凭 Claude 自身判断力决定何时写入。
3. **推荐的 A+C 组合**：Claude 负责在上下文中主动识别重要信息并写入 memory，用户随时可以用"记住这个"补充 Claude 漏掉的。
4. **暂缓实施**：用户因担心 memory 污染，决定在了解更多信息前不进行实际写入尝试。

## 第三部分：视频收录 + 对照验证

1. **收录了 ECC 系列 Hook 专集视频**到 wiki：创建 `[[ecc-hooks-video]]` source-note，更新 index 和 log。
2. **视频的理想化与现实的差距**：视频声称 PreCompact 能"保存关键状态"、Stop 能做"经验记录"——但 shell hook 根本做不到 AI 级判断。
3. **视频对代码项目更实用，对本 vault 需做减法**：Stop Hook 的"四件事"（格式化+类型检查+测试+经验记录）对本纯 Markdown 知识管理项目不适用。

## 第四部分：Hook 升级实施

1. **重新理解项目双用途**：这个 vault 不仅是笔记管理工具，也是学习 Claude Code 机制的实验场。
2. **PreCompact 和 Stop 从"时间戳"升级为"状态快照"**：抓取 wiki 篇数、outputs 篇数、最近操作日志。
3. **`>>` 改为 `>`（覆盖而非追加）**：避免 session-state.md 累积多轮垃圾。

**最终 Hook 配置结构：**

```
Stop → 捕获: wiki篇数 + outputs篇数 + outputs文件列表 + 最近5条日志 + /summary提醒
PreCompact → 捕获: wiki篇数 + outputs篇数 + outputs文件列表 + 最近3条日志
SessionStart → 加载并展示 session-state.md + macOS 通知
```

## 第五部分：outputs 纳入快照 + 最终状态评估

1. **outputs 比 log 更有上下文恢复价值**：log 告诉你"读了什么资料"，outputs 告诉你"产出了什么结论、做了什么决策"。
2. **最终状态评估：5 个合理点 + 1 个已承认的限制**——职责链路完整、信息结构合理、覆盖写入不累积垃圾、容错到位、没有越界。唯一限制是 shell 做不到 AI 级信息提取，属于固有边界。

## 第六部分：跨机器 Hook 部署方案

1. **Hook 配置集中在项目级 `.claude/settings.json`**，用户级 `~/.claude/settings.json` 是 API key 配置，无需复制。
2. **两种策略**：直接复制 JSON vs 自然语言指令。最终采纳后者——让另一台 Mac 的 Claude 自动探测 vault 结构并生成适配配置。
3. **自然语言指令的核心设计**：不描述具体文件路径，而是描述"做什么"和"为什么"，适应性更强。
4. **M 芯片兼容性确认**：macOS `osascript` 通知在 Intel/M 芯片间完全通用。

## 第七部分：/summary 命令两次优化

### 7.1 保存模式三选一

**原有问题**：`/summary` 直接覆盖当天已有文件，导致前次会话的总结内容丢失。

**改进**：保存前检查文件是否存在，若已存在则提供三个选择：
1. 合并 — 在原笔记基础上整合新旧内容
2. 新建 — 另存为 `{主题名}-{{date}}-N.md`
3. 覆盖 — 用新内容完全替换旧内容

### 7.2 主题名前缀

**原有问题**：文件名硬编码为 `对话总结-{{date}}.md`，多项目并行时所有 summary 往同一个文件里怼。

**改进**：保存前先询问本次总结的主题/项目名（如"投资策略""AI学习""Hook配置"），文件名格式 `{主题名}-{{date}}.md`。outputs/ 目录天然按项目/主题分区。

## 第八部分：多项目并行工作流

1. **SessionStart 快照不足以恢复多项目上下文**：`session-state.md` 用 `>` 覆盖写入，只保留最后一个关闭终端的状态。
2. **真正让 Claude 接上思路的是手动指定 outputs 文件**：新终端启动后发送「读取 outputs/XXX.md，继续推进」→ Claude 读取总结 → 重建认知状态。
3. **完整工作流**：关闭终端前 `/summary` 产出 `{主题名}-{日期}.md` → 下次启动时手动指定读取对应文件 → 继续推进。

## 关键概念

- **session-state.md**：hook 间通信协议，PreCompact/Stop 写入，SessionStart 读取。四层信息结构：时间戳 → outputs 列表 → 操作日志 → 提醒
- **outputs/ vs log.md**：outputs 是用户筛选的有效知识（/summary 产物），log 是机械的操作记录。outputs 上下文恢复价值远高于 log
- **自然语言配置指令**：用自然语言描述配置需求，让目标机器的 Claude 自己探索环境并生成适配的配置，而非复制硬编码文件
- **shell hook 的固有边界**：shell 做不到 AI 级信息提取，这是固有限制而非配置缺陷
- **主题名前缀**：`/summary` 文件名从固定变为 `{主题名}-{{date}}.md`，使 outputs/ 天然按项目分区

# 2026-05-22 追加：PermissionRequest 通知 hook

## 新增配置

在 hooks 中新增 `PermissionRequest` 事件，权限弹窗时触发：

```json
"PermissionRequest": [
  {
    "matcher": "",
    "hooks": [
      {
        "type": "command",
        "command": "afplay /System/Library/Sounds/Submarine.aiff && afplay /System/Library/Sounds/Submarine.aiff &"
      },
      {
        "type": "command",
        "command": "osascript -e 'display notification \"Claude 需要你的操作权限\" with title \"Claude 权限请求\"'"
      }
    ]
  }
]
```

**设计选择**：声音用 Submarine 而非 Glass（区分 Stop hook 的结束提醒），弹窗用 osascript 而非纯 afplay（确保有视觉通知）。

## 触发的经验教训：subagent 验证链断裂

本次配置过程中暴露了一个方法论问题：

1. **subagent 断言"只有 3 个 hook 事件"**（SessionStart/Stop/PreCompact），我直接采信
2. **用户两次纠正**：第一次说视频里见过，我仍在错误前提下提替代方案（iTerm2 触发器、扩 allowlist）；第二次坚持后我才去搜 web
3. **搜索结果立即证实有 PermissionRequest、PreToolUse、PostToolUse、Notification 等多个事件**
4. **最后还选错了**——在 PermissionRequest 和 Notification+permission_prompt 之间选了后者，直到用户用视频字幕纠正才回到正轨

**根因**：对 subagent 的排他性结论缺乏交叉验证习惯；用户纠正时不是重新审视前提，而是在错误框子里找替代路径。

已写入 memory（`feedback_verification.md`）：subagent 排他性结论须用 web 搜索交叉确认。

## 当前 Hook 全景
