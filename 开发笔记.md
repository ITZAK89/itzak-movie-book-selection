# 电影图书推荐 — 开发笔记

> 关联输出：[[05-resources/llm-wiki/outputs/电影图书推荐开发-2026-05-24|开发总结 (2026-05-24)]]

## 项目概述

聚合 Wikipedia / Wikidata 实时数据的电影和图书推荐网站。纯静态 HTML，file:// 直接打开。

## 访问方式

- 本地：`open "03-projects/电影图书推荐/index.html"`
- 本地 server：`python3 -m http.server 8766` → `http://localhost:8766/03-projects/电影图书推荐/index.html`
- 测试页：`test.html`（API 连通性验证）
- 诊断：浏览器 Console 运行 `runDiag()`（已移除，待恢复）
- 离线模式：`test-data.js`（导出后用 Export 按钮，gitignore）

## 架构决策

### 数据源

| 类型               | 来源                 | 方式                                             |
| ---------------- | ------------------ | ---------------------------------------------- |
| 电影 work 奖（7个）    | Wikidata SPARQL    | `?item wdt:P166 wd:Qxxx` + `wdt:P31 wd:Q11424` |
| 电影 person 奖（11个） | Wikidata SPARQL    | `?person wdt:P166 wd:Qxxx` + `pq:P1686 ?work`  |
| 图书（8个）           | Wikidata SPARQL    | work/person/person_simple 混合                   |
| 海报/封面            | Wikipedia REST API | `/page/summary/{title}` → `thumbnail.source`   |
| TSPDT            | JSON 快照            | 未实现                                            |
| 媒体书单             | JSON 快照            | LibraryThing CORS 不通，未实现                       |

### 去重策略

- `mergeFilms()`: 按 `normTitle(title)` 去重，跨年份合并，收集 `yearList` 和 `awards[]`
- work 查询加 `wdt:P31 wd:Q11424` 过滤，排除人物实体
- person 查询跳过无 `workLabel` 的行

### 缓存策略

| 层级 | TTL | Key |
|------|-----|-----|
| 单奖项 SPARQL | 24h | `award_{qid}` |
| 完整数据集 | 1h | `full_movies` / `full_books` |
| Wikipedia 摘要 | 7d | `wiki_enrich` |
| 版本号 | 永久 | `mbr_version`（变更时清除所有缓存） |

### 筛选 UI

- 两级：来源行 + 年份行
- 默认显示最新 10 部（All + All 年份）
- 选择具体来源或年份后不限数量

## 开发历程

### 阶段 1: API 验证
- test.html 验证 Wikipedia REST / Core / Wikidata SPARQL 连通性

### 阶段 2: 初版 HTML
- 单文件 HTML，内联 CSS + JS
- 数据加载、卡片渲染、弹窗、收藏
- 暗色/亮色主题

### 阶段 3: 缓存 & 去重
- localStorage 多层缓存 + 版本号清理
- 跨年份影片去重合并
- 两级筛选（来源→年份）

### 阶段 4: Q-ID 验证

- 逐个打开 Wikidata 页面，人工确认全部 18 个 Q-ID
- 修正：Grand Prix Q28455848→Q844804, Jury Prize Q1342432→Q164200
- 确认：Cannes Director Q510175, Actor Q586140, Actress Q840286 均正确
- 制作 qid-verify.html 方便后续验证

### 阶段 5: 数据方案反复——SPARQL 屡次失败

**SPARQL 初版遇到的问题**：
- person 模式 Q-ID 用错 → 部分奖项全返回空
- 复杂查询（person+work 联合）超时（>60s）
- 429 限流：连续查询被 Wikipedia 拒绝
- Q-ID 验证耗时：18 个 Q-ID 需逐个打开 Wikidata 人工确认

**第一次转向 Wikipedia — 失败**：
- 尝试解析每个电影节的总结页面（如 `2026_Cannes_Film_Festival`）
- 问题：每个电影节页面 HTML 结构完全不同（Cannes `award|winner|film`，Oscars `awards|film`）
- Venice/Berlin 页面用年份命名而非届数，URL 不可预测
- 429 限流同样存在
- **结论**：复杂度爆炸，放弃此路径

**关键转折 — 用户提出用奖项独立页面**：
- 每个奖项（金棕榈、金狮等）有独立 Wikipedia 页面（如 `Palme_d%27Or`）
- 页面内含完整历史获奖表格，一次 API 调用拿到全部数据（1990-）
- 表结构统一可预测，不需要猜测 URL
- **这是整个项目的转折点**：从"按电影节抓取当年获奖"改为"按奖项抓取全部历史"

## 排错记录：电影 Wikipedia 解析

> 此部分根据 git log 回溯重建，具体调试细节有限。

### Oscar 获奖者 vs 提名者混淆
- **现象**：Oscar 表格中提名者和获奖者混在一起
- **根因 1**：Oscar 行有 `rowspan` 标记，获奖行 `rowspan>=3`，提名行被 rowspan 继承年份但本身不是获奖者
- **解法 1**：检测 `rowspan >= 3` 的行跳过（仅标记获奖者行）
- **根因 2**：仅靠 rowspan 不够，有些年份获奖者无 rowspan
- **解法 2**：加 `hasWinnerMarking` 检测——扫描前几行查找 `style="background:#FAEB86"`，有此标记的表才过滤非获奖行

### Oscar 年份格式
- **现象**：Oscar 年份格式 `1990(63rd)` 导致年份正则匹配失败
- **解法**：正则改为 `/^(\d{4})/`（只提取前四位数字）

### 电影列检测误判
- **现象**：表头含 "year of film" 时，`filmCol` 检测到 "film" 子串但实际是年份列
- **解法**：`filmCol` 检测排除含 "year" 的列（`ht.indexOf('film') >= 0 && ht.indexOf('year') < 0`）

### Wikipedia 搜索海报错误
- **现象**：用 Wikipedia Search API 找电影海报，命中演员页而非影片页
- **解法**：从 film 单元格 `<a href="/wiki/...">` 直接提取 Wikipedia 页面标题，100% 准确，不再依赖搜索

### Cannes Jury Prize 页面 0 items
- **现象**：Q-ID Q1342432 的 SPARQL 返回空
- **根因**：Q-ID 错误，实际为 Q164200
- **解法**：修正 Q-ID（后续全部迁移到 Wikipedia 页面解析，不再用 Q-ID）

### 语法错误
- **现象**：`index.html` 报 SyntaxError
- **根因**：编辑时误留下重复的 `];`
- **教训**：反复 Edit 大 HTML 容易引入语法错误 → 后续用 Python 脚本做替换

### 缓存策略调整
- **现象**：24h TTL 后数据自动过期，用户需要点刷新
- **决策**：缓存永不过期，仅手动刷新或版本号变化时重取

## 排错记录：图书 Wikipedia 解析

### Multi-line Edit 工具失效
- **现象**：Edit 工具对多行文本始终报 "String to replace not found"
- **根因**：Read 工具展示的内容与实际文件 whitespace 不一致（tab vs space 等不可见字符差异）
- **解法**：改用 Python 脚本（`python3 << 'PYEOF'`）做批量替换，直接操作文件字符串
- **教训**：大 HTML 文件的复杂修改不要依赖 Edit 工具，用脚本定位+替换更可靠

### Nobel Literature 解析失败
- **现象 1**：page=`Nobel_Prize_in_Literature` 返回 0 items
- **根因 1**：API 只返回了统计表（year|total|nominees|...），laureates 表在 `List_of_Nobel_laureates_in_Literature`
- **解法 1**：纠正页面名
- **现象 2**：换页后 parseBookTable 找到 laureates 表但 `authorCol` 指向 Image 子列，人名丢失
- **根因 2**：表头有两行 — Row0 `Year|Laureate(colspan=2)|Country|Citation`，Row1 `Image|Name`。parseBookTable 只读 Row0，`authorCol=1` 指向 colspan 的 Image 位置
- **解法 2**：检测 subHeaders（Row1 th），匹配 "name" 子列，`authorCol = headerCol + subHeaderIndex`。同时修复 sub-header 检测条件（`>= headerCells.length` → `> 0`）

### Intl Booker 国旗图片
- **现象**：2005-2015 年卡片显示国旗而非作者照片
- **根因**：该时段表无 title/work 列（year|author|country|translator|language|ref），`titleCol` 回退到 index 2（country 列，含国旗 `<img>`），Wikipedia 链接指向国家页面
- **解法**：当 titleCol 指向的单元格含 `<img>` 时，跳过而非搜索其他列。title 为空+author 有值时自动按 person 处理（`title=author, linkFromAuthor=true`），链接指向作者页
- **额外教训**：初版加了 image-fallback 搜索（遍历其他列找 title），结果找到 translator/language 列产生更差的数据。删掉 fallback，直接走 person 回退更干净

### National Book Award 2025 数据缺失
- **现象 1**：2025 年获奖数据完全不显示
- **根因 1**：2025 行使用 `<th rowspan="6">2025</th>` 格式（独占一行），不同于以往的 `<td>year</td>`
- **解法 1**：加 `cfg.rowspanYear` 标志，检测单 cell+TH+4位数字的行，提取 `pendingYear`+`pendingRowspan`，后续行自动继承年份
- **现象 2**：2025 数据显示了但列错位（"Winner" 出现在作者位置）
- **根因 2**：rowspan 年份不占数据行的单元格，实际列比配置少一列（yearCol 被 rowspan th 吞噬）。`authorCol=1` 实际指向 `cells[0]`
- **解法 2**：当 `pendingYear` 激活时，`effAuthorCol = authorCol - 1`，其他列同理

### National Book Award 2025 显示提名者
- **现象**：2025 年显示 6 条（1 winner + 5 finalists），其他年份只显示 winner
- **根因**：2025 表包含完整 shortlist，获奖行用 `style="background:LemonChiffon;"` 标记
- **解法**：加 `cfg.winnerBg` 标志，检测 row 或 cell 的 style 属性含 "background"，非获奖行跳过（类似 Oscar 的 `hasWinnerMarking` 逻辑）

### Women's Prize 返回 0 items
- **现象**：page=`Women's_Prize_for_Fiction` 找不到任何表格
- **根因**：页面名错误，实际页面为 `List_of_Women's_Prize_for_Fiction_winners`
- **解法**：纠正页面名

### 自动列检测 → 配置驱动
- **初衷**：自动识别列名适配不同表格结构
- **问题**：每个奖项的表格结构差异太大（两级表头、国旗列、rowspan 年份等），auto-detection 打补丁越来越复杂
- **决策**：每个奖项在 `BOOK_PAGES` 中显式指定 `yearCol/authorCol/titleCol`，加 `subAuthor/rowspanYear/winnerBg` 等标志位。解析器不再猜列，直接读配置
