

# 电影图书推荐 — Spec

## 目标

聚合权威外部来源的实时推荐数据，一站式浏览"被权威机构推荐过的"电影和（有中译本的）图书。不建本地内容库，数据全部从 API 实时拉取。纯静态 HTML，浏览器 file:// 直接打开即用。

## 核心理念

**"推荐"不是我们自己判断，而是聚合权威来源的推荐信号：**
- 电影：国际电影节获奖 = 推荐，TSPDT 千佳排名 = 推荐
- 图书：国际文学奖获奖/入围 = 推荐（原版语言不限，只要有中译本或大概率有中译本）

## 数据源清单

### 电影

| 数据源 | 内容 | 获取方式 | CORS | 更新频率 |
|--------|------|----------|------|----------|
| Wikipedia REST API | 电影节页面摘要（2026 Cannes, 2026 Venice 等） | `fetch()` GET | ✅ | 实时 |
| Wikipedia Core API | 解析电影节获奖表格 | `fetch()` + `origin=*` | ✅ | 实时 |
| Wikidata SPARQL | 结构化查询：某电影节某年份获奖影片+导演 | `fetch()` GET | ✅ | 实时 |
| TSPDT (gf1000.htm) | 影史千佳排名 | ❌ 浏览器无法直连，**预生成 `tspdt.json` 快照** | — | 每年 1 月手动更新一次快照 |

**覆盖的电影节/奖项：**
- 戛纳 (Cannes)：金棕榈、评审团大奖、最佳导演、最佳男女演员
- 威尼斯 (Venice)：金狮奖
- 柏林 (Berlin)：金熊奖
- 奥斯卡 (Oscars)：最佳影片、最佳国际影片
- TSPDT 1000 佳（年度快照补充）

### 图书

| 数据源 | 内容 | 获取方式 | CORS | 更新频率 |
|--------|------|----------|------|----------|
| Wikipedia Core API | 解析文学奖获奖/入围表格 | `fetch()` + `origin=*` | ✅ | 实时 |
| Wikidata SPARQL | 结构化查询：获奖图书+中译本信息 | `fetch()` GET | ✅ | 实时 |

**覆盖的图书奖项（有中译本概率高）：**
- 布克奖 (Booker Prize)
- 国际布克奖 (International Booker Prize)
- 普利策小说奖 (Pulitzer Prize for Fiction)
- 诺贝尔文学奖 (Nobel Prize in Literature)
- 美国国家图书奖 (National Book Award)
- 女性小说奖 (Women's Prize for Fiction)
- 芥川奖 / 直木奖（日本，中译本概率极高）

**中译本判断策略：**
- 优先通过 Wikidata 查询 `has edition or translation (P747)` → 中文 (Q7850)
- 若 Wikidata 无中译本记录，仍展示但标注"中译本待确认"
- 国际重要奖项获奖作品默认假设有中译本，不做硬过滤

> 注意：中国本土文学奖（茅盾奖等）也纳入，因为它们是中文原版书，天然满足"可阅读"条件。通过 Wikidata 查询覆盖。

### 海报与封面图

**来源：Wikipedia REST API `/page/summary/{title}`**

该端点对每条影片/图书返回：
```json
{
  "title": "Fjord (film)",
  "thumbnail": { "source": "https://...jpg", "width": 320 },
  "originalimage": { "source": "https://...jpg", "width": 2000 }
}
```
- 卡片网格用 `thumbnail`（小图，减少加载时间）
- 详情弹窗用 `originalimage`
- Wikipedia 无封面图的情况：显示 CSS 渐变占位图（片名/书名居中叠加）

**数据流中的位置：**
```
Wikidata SPARQL → [影片/图书列表 + Wikipedia 页面标题]
       ↓
对每个 title 调 Wikipedia REST API /page/summary/{title}
       ↓ 拿到:
       ├─ extract (简介文字)
       ├─ thumbnail.source (海报/封面小图)
       └─ originalimage.source (海报/封面大图)
       ↓
            统一数据模型 → 渲染卡片
```

**性能考虑：**
- Wikidata SPARQL 一次返回约 20-50 条结果
- 每条结果需单独调用 Wikipedia REST API 取封面
- 策略：**先渲染卡片骨架（无图），图片异步加载到位后填充**
- 并发控制：最多同时 6 个 fetch（浏览器默认并发限制），不必额外节流

## 容错与缓存策略

```
页面加载
  ├─ 尝试 fetch Wikipedia API / Wikidata
  │   ├─ 成功 → 展示数据 + 写入 localStorage cache
  │   └─ 失败 → 读取 localStorage cache
  │       ├─ 有缓存 → 展示缓存数据 + 顶部 banner "⚠️ 实时数据更新失败，显示上次缓存 (2026-05-20)"
  │       └─ 无缓存 → 展示空状态 "无法加载数据，请检查网络后刷新"
  └─ TSPDT 数据
      ├─ 加载预置 tspdt.json（本地文件，始终可用）
      └─ 不依赖网络
```

## 技术选择

| 决策 | 选择 | 原因 |
|------|------|------|
| 框架 | 无，纯 vanilla HTML/CSS/JS | 零依赖，file:// 即用 |
| CSS | 内联 `<style>` | 单文件自包含 |
| 数据获取 | `fetch()` 调用 Wikipedia/Wikidata API | CORS 友好，无需 API Key |
| TSPDT 数据 | 预生成 `tspdt.json`，`<script>` 标签引入 | TSPDT 站点无 CORS，浏览器无法直连 |
| 持久化 | localStorage | 缓存 API 响应 + 收藏夹 |
| 构建 | `regenerate_html.py` 从 JSON + 模板生成 HTML | 避免手工编辑大型 HTML |
| TSPDT 快照更新 | Python 脚本 `fetch_tspdt.py` | 手动/定时抓取 gf1000.htm，解析为 JSON |

### 已知数据问题与处理策略

| 问题 | 策略 |
|------|------|
| SPARQL 返回导演+影片两条重复 | JS 按 `filmLabel` 去重，优先保留影片实体 |
| 金狮/金熊/诺贝尔/国际布克 Q-ID 待二轮验证 | test.html 已修复 Q-ID，刷新页面确认 |
| 新片/冷门片 Wikipedia 无封面 | CSS 渐变占位图降级 |
| 中译本 SPARQL 查询超时 (65s) | **不做逐条验证，默认国际重要奖项获奖作品有中译本** |

### 中译本策略（最终决定）

**默认所有国际文学奖获奖/入围作品有中译本，不做逐条 Wikidata 验证。**

理由：
- Wikidata 中译本（P747→Q7850）查询涉及聚合+JOIN，实际超时 65s
- 布克奖/诺贝尔/普利策/国际布克等级别的作品，中译本覆盖率接近 100%
- 个别冷门作品无中译本是可接受的误差
- 可在 UI 标注"中译本状态：假定有（未逐条验证）"

## 已知风险

- Wikipedia 页面结构可能变化（获奖表格的 HTML class/markup 不保证稳定）
- Wikidata SPARQL 查询可能超时（复杂查询限制 30s），需拆分查询
- TSPDT 每年更新一次，快照可能过期，需在 UI 标注快照日期
- 图书的中译本判断不完美——Wikidata 的中译本数据可能不完整，最终是"最佳努力"策略
- 部分影片/图书在 Wikipedia 上没有封面图（尤其是较冷门的作品），需设计占位图方案
- Wikipedia REST API 封面图需逐条单独请求，50 条结果 = 50 次额外 fetch，页面冷启动时图片加载有延迟
